# 互联网项目压力测试指南

> 定位：压测是 Sentinel 限流阈值配置的数据依据，也是系统上线前的必要环节。

---

## 一、压测的目的

压测不是为了让系统崩溃，而是为了找到三条线：

```
系统容量边界
    ├── 最佳性能点（Best Performance Point）
    │       在这个 QPS 下，响应时间最短，吞吐量最优
    ├── 最大容量点（Max Throughput Point）
    │       再往上加压，吞吐量不再增加，请求开始排队
    └── 崩溃点（Breaking Point）
            线程池耗尽，响应时间急剧上升，开始大量报错
```

**Sentinel 的限流阈值就设在最大容量点附近**，超过这个值直接拒绝请求，保护系统不到达崩溃点。

## 二、常用压测工具

### JMeter（最主流）

Apache 出品，GUI 界面，功能最全，电商项目使用最广泛。

```
优点：图形化配置、插件丰富、支持分布式压测、报告完善
缺点：高并发时 JMeter 本身消耗资源较大，影响压测结果准确性
适合：功能测试 + 压力测试 + 接口测试，团队协作场景
```

### Gatling（代码驱动）

用 Scala DSL 编写测试脚本，性能比 JMeter 好，报告更直观漂亮。

```scala
// Gatling 脚本示例：压测推荐品牌接口
val scn = scenario("RecommendList压测")
  .exec(http("获取推荐品牌")
    .get("/portal/brand/recommendList")
    .queryParam("pageNum", "1")
    .queryParam("pageSize", "6"))

setUp(
  scn.inject(
    rampUsers(1000).during(60.seconds)  // 60秒内爬坡到1000并发
  )
).protocols(httpProtocol)
```

```
优点：资源消耗小、报告漂亮、脚本可纳入版本控制
缺点：需要会 Scala，学习成本稍高
适合：性能要求高的场景，DevOps 流水线集成
```

### wrk / wrk2（命令行）

轻量级工具，适合快速验证单接口性能，无需安装复杂环境。

```bash
# 用 8 个线程、500 个并发连接压测 30 秒
wrk -t8 -c500 -d30s http://localhost:8088/portal/brand/recommendList

# 输出示例：
# Requests/sec: 8523.45   ← QPS
# Latency avg: 58.32ms    ← 平均响应时间
# Transfer/sec: 2.14MB
```

```
优点：极轻量、结果快速、资源消耗极低
缺点：不支持复杂场景（登录态、混合流量）
适合：单接口快速基准测试
```

### 云端压测平台

| 平台 | 提供商 | 特点 |
|------|-------|------|
| PTS | 阿里云 | 支持全链路压测，和阿里云产品深度集成 |
| WeTest | 腾讯云 | 支持 App 和 Web |
| LoadRunner | MicroFocus | 老牌企业级工具，授权费高 |

适合需要模拟大规模并发（万级以上）且不想自己维护压测集群的场景。

## 三、标准压测流程

### 第一步：明确压测目标

不是盲目压测，先定好量化目标：

```
目标示例：
- recommendList 接口 P99 响应时间 < 200ms
- 系统 QPS 达到 500 时错误率 < 0.1%
- 大促期间峰值 QPS 预估 2000，留 30% 余量，压测目标 QPS 2600
```

**P99 的含义**：99% 的请求响应时间都在这个值以内，比平均响应时间更能反映用户真实体验，因为平均值会掩盖少量极慢的请求。

### 第二步：构造压测场景

电商压测不只压单接口，要模拟真实用户行为的混合流量：

```
典型电商压测流量模型：
    60% 浏览商品（GET，无鉴权，可缓存，读多）
    20% 搜索商品（GET，Elasticsearch 压力）
    10% 加入购物车（POST，有鉴权，写库）
     5% 下单（POST，有鉴权，写库 + 扣库存，事务）
     5% 支付（POST，高价值接口，需要特殊保护）
```

用混合流量压测，结果比单接口压测更接近真实场景，能暴露接口间的资源竞争问题（如数据库连接池争用）。

### 第三步：爬坡加压

不要一上来就最大并发，要逐步增加，观察系统在各压力级别的表现：

```
阶段一：基准测试
    10 并发 → 记录基准响应时间和 QPS

阶段二：逐步加压
    100 QPS → 观察响应时间变化
    500 QPS → 观察响应时间和错误率
    1000 QPS → 寻找瓶颈点

阶段三：极限测试
    持续加压直到错误率明显上升 → 记录崩溃点

阶段四：稳定性测试
    在最大容量点的 70% 处持续压测 1-2 小时
    验证系统在高负载下的长期稳定性（排查内存泄漏等问题）
```

### 第四步：观察关键指标

压测期间需要同时监控多个维度的指标：

| 指标 | 说明 | 需要关注的阈值 |
|------|------|-------------|
| QPS | 每秒处理请求数 | 低于预期说明存在瓶颈 |
| 响应时间 P50/P95/P99 | 各百分位响应时间 | P99 超过 500ms 需要优化 |
| 错误率 | 4xx/5xx 请求比例 | 超过 0.1% 需要立即排查 |
| CPU 使用率 | 服务器 CPU 占用 | 持续超过 70% 需要扩容 |
| JVM Heap 使用率 | Java 堆内存占用 | 持续接近上限说明内存不足 |
| Full GC 频率 | JVM 老年代回收频率 | 频繁 Full GC 会造成停顿 |
| 数据库连接池 | 活跃连接数 / 最大连接数 | 接近上限说明 DB 是瓶颈 |
| Tomcat 线程池 | 活跃线程数 / 最大线程数 | 接近上限说明处理能力不足 |
| Redis 命中率 | 缓存命中比例 | 低于 90% 说明缓存策略有问题 |

### 第五步：定位瓶颈

压测结果出来后，瓶颈一般出现在几个固定位置：

**接口响应慢：**
```
数据库慢查询   → 添加索引、优化 SQL、开启慢查询日志定位
缺少缓存       → 热点数据加 Redis 缓存
外部调用慢     → 设置合理超时时间、异步化非关键调用
代码热点       → 使用 Arthas 或 async-profiler 做 CPU Profiling
```

**系统崩溃：**
```
Tomcat 线程池耗尽  → 增大线程池上限 或 加 Sentinel 限流
数据库连接池耗尽   → 增大连接池 或 引入读写分离
JVM 内存溢出       → 调整堆大小 或 排查内存泄漏（MAT 分析 heap dump）
网络带宽打满       → CDN 分流静态资源 或 扩容带宽
```

## 四、压测与 Sentinel 的关系

压测结果直接指导 Sentinel 限流阈值的配置，让限流有数据支撑而不是拍脑袋：

```
压测结论示例：
    mall-admin recommendList 接口
    最佳性能点：QPS = 500（响应时间最低，约 30ms）
    最大容量点：QPS = 800（吞吐量不再增加，响应时间开始上升）
    崩溃点：     QPS = 1200（错误率超过 5%）

Sentinel 配置：
    限流阈值 = 800 × 70% = 560 QPS（最大容量的 70%，留安全余量）
    超过 560 QPS 直接返回降级结果

最终效果：
    正常流量：所有请求正常处理，响应时间稳定
    突发流量：超额请求被 Sentinel 快速拒绝（返回降级结果）
    系统始终运行在安全区间，不会因流量突增而崩溃
```

**为什么设最大容量的 70% 而不是 100%？**

压测环境和生产环境存在差异（机器规格、网络、其他服务竞争资源），留 30% 余量是工程经验，防止生产环境实际容量低于压测结论。

## 五、大厂压测实践：阿里全链路压测

阿里双十一的全链路压测是业界最典型的案例，核心思路是**影子流量**：

```
压测请求打上特殊标记（Header: X-Pressure-Test: true）
    ↓
流量走真实的生产链路（Nginx → Gateway → 业务服务）
    ↓
写操作识别压测标记，落到影子数据库（不污染真实数据）
读操作从影子缓存读取（不影响真实缓存）
    ↓
压测结果完全反映生产环境真实性能
```

**优点**：压测结果和生产一致，发现的问题都是真实的生产问题。

**缺点**：技术复杂度极高，需要所有中间件（DB、Redis、MQ）都支持影子流量隔离，实现成本巨大。

中小公司一般做法是维护独立的**预发布环境**（配置和生产一致，数据量用生产数据的子集），在预发布环境做压测，虽然不如全链路压测精确，但实现成本可接受。

## 六、对 mall 项目的实践建议

### 快速入门：用 wrk 做基准测试

```bash
# 第一步：压测公开接口（无需 Token）
wrk -t4 -c100 -d30s \
  "http://localhost:8088/portal/brand/recommendList?pageNum=1&pageSize=6"

# 第二步：压测需要鉴权的接口（先登录拿 Token）
wrk -t4 -c100 -d30s \
  -H "Authorization: Bearer eyJhbGci..." \
  "http://localhost:8088/admin/brand/list?pageNum=1&pageSize=10"
```

### 根据结果配置 Sentinel 阈值

```
记录 wrk 输出的 Requests/sec（QPS）
    ↓
Sentinel 限流阈值 = QPS × 70%
    ↓
在 Sentinel 控制台或 Nacos 配置文件中设置
```

### 关注的核心接口（优先级从高到低）

| 接口 | 原因 |
|------|------|
| /portal/brand/recommendList | 首页高频接口，流量最大 |
| /portal/home/content | 首页内容，依赖多个下游 |
| /admin/admin/login | 防止暴力破解 |
| /portal/order/generateOrder | 下单接口，涉及库存扣减 |

## 七、总结

```
压测工具选择
    ├── 快速验证单接口    → wrk（最简单）
    ├── 功能 + 性能测试   → JMeter（最通用）
    ├── 代码化压测脚本    → Gatling（DevOps 友好）
    └── 万级并发压测      → 云端平台（PTS/WeTest）

压测流程
    明确目标 → 构造混合场景 → 爬坡加压 → 监控指标 → 定位瓶颈

压测结果的应用
    最大容量点 × 70% = Sentinel 限流阈值
    崩溃点 × 50% = 扩容触发阈值
    P99 响应时间 = 接口 SLA 基准
```
